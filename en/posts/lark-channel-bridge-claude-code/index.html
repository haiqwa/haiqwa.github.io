Connecting Feishu to Claude Code with lark-channel-bridge: AI Pair Programming Anywhere

2026-06-07 16:00:00·Vibe Coding·#Claude Code#Feishu#Lark#Agent#Vibe Coding

The Trigger: Wanting to Fix a Bug While Away from My Desk

As a heavy Claude Code user, my biggest pain point isn't that the model isn't smart enough — it's that I have to be sitting at my computer to use it.

One day I was out and suddenly thought of a layout issue on my blog that I wanted to fix. I opened Feishu to tell a colleague about it, and then thought: what if I could just talk to Claude Code directly in Feishu? When I got home and checked, sure enough, someone had already built this.

The core idea behind lark-channel-bridge is beautifully simple: bridge Feishu messages to a locally running Claude Code CLI. You send a message in Feishu, the bridge passes it to your local Claude Code process; Claude's reply gets sent back to Feishu through the bridge. Just like that, your phone becomes a remote control for Claude Code.

Architecture: A Message's Journey

The entire system has only three components:

Feishu App ←→ Feishu Open Platform (WebSocket) ←→ lark-channel-bridge ←→ Claude Code CLI
  1. Feishu Open Platform: You create a Feishu app, enable the bot capability, and configure event subscriptions. Feishu pushes user messages to the bridge via WebSocket.
  2. lark-channel-bridge: A Node.js process running on your local machine. It maintains a persistent WebSocket connection with Feishu, receives messages, and spawns a Claude Code subprocess to handle them.
  3. Claude Code CLI: The familiar claude command-line tool. The bridge starts it in SDK mode, passes in the user message, and gets back the reply.

The key design decision: the bridge runs locally, not in the cloud. This means Claude Code can directly access your filesystem, Git repos, and local tools — exactly the same as when you use it in the terminal. The tradeoff is that your computer needs to be on, but you get zero-config local environment access in return.

Setup Process

Step 1: Create a Feishu App

Create a custom app on the Feishu Open Platform:

  1. Enable the Bot capability
  2. Add im.message.receive_v1 (receive message) in event subscriptions
  3. Note down the App ID and App Secret
  4. Publish the app and get admin approval

💡 If you're only using it personally, you can add yourself as the app admin in the Feishu admin console, so you don't need organization admin approval.

Step 2: Install lark-channel-bridge

# Clone the repository
git clone https://github.com/anthropics/lark-channel-bridge.git
cd lark-channel-bridge

# Install dependencies
npm install

# Configure
cp .env.example .env
# Edit .env, fill in the Feishu app's App ID, App Secret, etc.

Step 3: Start the bridge

npm run start

If everything goes well, you'll see logs showing the bridge successfully connected to the Feishu WebSocket. Now open Feishu, find your bot, and send a message to test.

Pitfalls I Encountered

Windows Path Issues

My development environment is Windows, and some path handling in the bridge assumes Unix-style paths. For example, working directory passing uses D:\homepage on Windows while the bridge expects /d/homepage. The solution is to explicitly specify the working directory at startup and ensure Claude Code's project path is set correctly.

Bots in Groups Only Receive Messages When @'d

This is a Feishu platform limitation, not a bug. Bots in group chats only receive messages when they are genuinely @'d (structured mention). Typing "@botname" in plain text doesn't work. This means if you add the bot to a project group, team members need to develop the habit of @'ing it.

OAuth Authorization Can't Be Done in Groups

The bridge supports letting Claude Code use user identity (rather than app identity) to call Feishu APIs, which requires OAuth authorization. But OAuth's device flow generates a verification link — if you post this link in a group, whoever clicks it first gets the token, binding it to the wrong identity.

The correct approach: initiate authorization only in private chats (p2p). The bridge automatically detects the chat type and refuses to execute lark-cli auth login in group chats.

Background Processes Get Recycled

The bridge reclaims the Claude Code subprocess after your reply is complete. If you start lark-cli auth login with run_in_background: true, this background process gets killed along with the subprocess. So the authorization flow must run in foreground blocking mode — first use --no-wait to get the verification link, then use --device-code to block and wait for the user to complete authorization.

Advanced Usage

Interactive Cards: More Than Just Text Replies

The bridge supports sending Feishu interactive cards (CardKit 2.0 schema), which means you can have Claude return rich interactive interfaces with buttons and forms.

For example, have Claude send you a selection card:

You: Help me deploy the blog
Claude: [Sends a card with "Preview" and "Production Deploy" buttons]
You: [Click "Production Deploy"]
Claude: Got it, starting deployment...

Card callbacks are verified through bridge_token signatures, ensuring click events return to the same Claude session for continued processing.

Multi-Bot Collaboration

The bridge supports pulling multiple different AI bots into the same group and enabling collaboration through the @ mechanism:

  • @ Claude Code for code tasks
  • @ another bot for translation/summarization
  • Even have bots @ each other to complete complex workflows

But be careful to avoid "ping-pong loops" — two bots @'ing each other infinitely. The bridge's rule is: by default, don't @ other bots; only hand off when the user explicitly requests it.

Integration with Claude Code Skills

Because the bridge is essentially Claude Code underneath, all the skills, workflows, and CLAUDE.md rules you've configured locally are fully available on the Feishu side.

My blog project has skills configured like /check-post for validating blog format and /fetch-zhihu for scraping Zhihu articles. Now I can send /fetch-zhihu https://zhuanlan.zhihu.com/p/xxx to the Feishu bot from my phone, and it will scrape the article, validate the format, and auto-fix issues — exactly the same as if I were sitting at my computer.

Security Considerations

Exposing your local Claude Code through an instant messaging tool requires serious security consideration. First, access control: the bridge controls access through Feishu identity verification, so only messages from people in designated chats are processed. Second, permission boundaries: Claude can do the same things as in the terminal — it's constrained by your project's CLAUDE.md rules, and Claude Code itself has a permission confirmation mechanism. Finally, data flow: messages pass through Feishu servers and the bridge to reach local Claude Code; code execution happens entirely on your machine, and nothing is uploaded to additional third-party services.

The Result

My daily workflow now looks like this:

  • 📱 When I'm out, I message the bot on Feishu to fix bugs, write articles, or deploy the website
  • 💻 When I get back to my computer, all changes are already in local files
  • 🔄 Seamless multi-device handoff, conversation context persists

That feeling of "having a full-stack engineer on call anywhere, anytime" is honestly a bit addictive.

Conclusion

The value of lark-channel-bridge goes beyond just "putting Claude Code on your phone." What it really does is turn an instant messaging tool into a unified entry point for AI Agents. Feishu is the core of the work scenario; Claude Code is the core of the development scenario. The bridge connects the two — you execute where you communicate.

If you're also using Claude Code, I highly recommend giving it a try. Set it up once, use it for a long time.
