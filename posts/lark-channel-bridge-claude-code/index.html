用 lark-channel-bridge 把飞书接上 Claude Code：随时随地 AI Pair Programming

2026-06-07 16:00:00·Vibe Coding·#Claude Code#飞书#Lark#Agent#Vibe Coding

起因：蹲坑时想改个 bug

作为一个重度 Claude Code 用户，我最大的痛点不是模型不够聪明，而是——我得坐在电脑前才能用它。

某天在外面，突然想到博客有个排版问题想修。打开飞书想跟同事说一声，然后想：如果我能直接在飞书里跟 Claude Code 说话就好了。回家一看，果然有人做了这件事。

lark-channel-bridge 的核心思路非常简单：把飞书消息桥接到本地运行的 Claude Code CLI。你在飞书里发一条消息，bridge 把它转给本地的 Claude Code 进程；Claude 的回复再通过 bridge 发回飞书。就这样，你的手机变成了 Claude Code 的遥控器。

架构：一条消息的旅程

整个系统只有三个组件：

飞书 App ←→ 飞书开放平台(WebSocket) ←→ lark-channel-bridge ←→ Claude Code CLI
  1. 飞书开放平台：你创建一个飞书应用，开启机器人能力，配置事件订阅。飞书通过 WebSocket 把用户消息推送给 bridge。
  2. lark-channel-bridge：跑在你本地机器上的一个 Node.js 进程。它维护与飞书的 WebSocket 长连接，接收消息后启动一个 Claude Code 子进程来处理。
  3. Claude Code CLI：就是大家熟悉的 claude 命令行工具。bridge 用它的 SDK 模式启动，传入用户消息，拿回回复。

关键设计决策：bridge 不在云端跑，在你本地跑。这意味着 Claude Code 能直接访问你的文件系统、Git 仓库、本地工具——跟你在终端里用完全一样。代价是你的电脑得开着，但换来的是零配置的本地环境访问。

搭建过程

第一步：创建飞书应用

在飞书开放平台创建一个自建应用：

  1. 开启机器人能力
  2. 在事件订阅里添加 im.message.receive_v1（接收消息）
  3. 记下 App ID 和 App Secret
  4. 发布应用，获取管理员审批

💡 如果你只是个人使用，可以在飞书管理后台把自己加为应用管理员，这样就不需要组织管理员审批。

第二步：安装 lark-channel-bridge

# 克隆仓库
git clone https://github.com/anthropics/lark-channel-bridge.git
cd lark-channel-bridge

# 安装依赖
npm install

# 配置
cp .env.example .env
# 编辑 .env，填入飞书应用的 App ID、App Secret 等

第三步：启动 bridge

npm run start

如果一切正常，你会看到 bridge 成功连接飞书 WebSocket 的日志。现在打开飞书，找到你的机器人，发一条消息试试。

踩坑记录

Windows 路径问题

我的开发环境是 Windows，bridge 里面有些路径处理假设是 Unix 风格。比如工作目录的传递，Windows 用 D:\homepage 而 bridge 期望 /d/homepage。解决方法是在启动时显式指定工作目录，并且确保 Claude Code 的项目路径设置正确。

群聊里 bot 只能收到 @ 消息

这是飞书的机制限制，不是 bug。机器人在群聊里只有被真实 @（结构化 mention）才能收到消息。纯文本写"@机器人名字"是不行的。这意味着如果你把 bot 拉进一个项目群，团队成员需要养成 @ 它的习惯。

OAuth 授权不能在群里做

bridge 支持让 Claude Code 使用用户身份（而非应用身份）调用飞书 API，这需要 OAuth 授权。但 OAuth 的 device flow 会生成一个验证链接——如果你在群里发这个链接，谁先点谁就拿走 token，会绑定到错误的身份。

正确做法是：在私聊（p2p）里发起授权。bridge 会自动检测聊天类型，群聊里拒绝执行 lark-cli auth login。

后台进程会被回收

bridge 在你回复结束后会回收 Claude Code 子进程。如果你用 run_in_background: true 启动了 lark-cli auth login，这个后台进程会跟着子进程一起被杀掉。所以授权流程必须用前台阻塞的方式跑——先 --no-wait 拿到验证链接，再 --device-code 阻塞等待用户完成。

进阶玩法

交互卡片：不只是文字回复

bridge 支持发送飞书交互卡片（CardKit 2.0 schema），这意味着你可以让 Claude 返回带按钮、表单的富交互界面。

比如，让 Claude 给你发一张选择卡片：

你：帮我部署博客
Claude：[发送一张卡片，上面有"预览"和"正式部署"两个按钮]
你：[点击"正式部署"]
Claude：收到，开始部署...

卡片回调通过 bridge_token 签名验证，确保点击事件能回到同一个 Claude 会话继续处理。

多 Bot 协作

bridge 支持把多个不同的 AI bot 拉进同一个群，通过 @ 机制实现协作：

  • @ Claude Code 处理代码任务
  • @ 另一个 bot 做翻译/摘要
  • 甚至让 bot 之间互相 @ 完成复杂工作流

但要注意避免"乒乓死循环"——两个 bot 互相 @ 无限对话。bridge 的规则是：默认不 @ 其他 bot，只有用户明确要求时才转交。

与 Claude Code Skills 联动

因为 bridge 底层就是 Claude Code，所以你在本地配置的所有 skills、workflows、CLAUDE.md 规则在飞书端都完全可用。

我的博客项目配置了 /check-post 校验博客格式、/fetch-zhihu 抓取知乎文章等 skill。现在在手机上给飞书 bot 发一条 /fetch-zhihu https://zhuanlan.zhihu.com/p/xxx，它就会抓取文章、校验格式、自动修复问题——跟我坐在电脑前操作一模一样。

安全考量

把本地的 Claude Code 暴露到即时通讯工具上，安全性需要认真对待。首先是访问控制：bridge 通过飞书身份验证控制访问，只有指定聊天中的人发的消息才会被处理。其次是权限边界：Claude 能做什么跟终端里一样，它受你项目的 CLAUDE.md 规则约束，也有 Claude Code 本身的权限确认机制。最后是数据流向：消息经过飞书服务器和 bridge 到达本地 Claude Code，代码执行完全在你的机器上，不会上传到额外的第三方服务。

最终效果

现在我的日常工作流是这样的：

  • 📱 在外面用飞书给 bot 发消息，让它改 bug、写文章、部署网站
  • 💻 回到电脑前，所有改动已经在本地文件里了
  • 🔄 多设备无缝衔接，对话上下文持续保持

那种"随时随地有个全栈工程师等着你使唤"的感觉，确实有点上瘾。

总结

lark-channel-bridge 的价值不只是"把 Claude Code 搬到手机上"。它真正做的是把即时通讯工具变成了 AI Agent 的统一入口。飞书是工作场景的核心，Claude Code 是开发场景的核心，bridge 把两者连起来——你在哪里沟通，就在哪里执行。

如果你也在用 Claude Code，强烈建议试试。搭一次，用很久。
